ADR-001: Why SQLite is NOT Used in the MVP
Status: Accepted
Context
The initial project plan considered using SQLite as a storage layer for the AM4 datasets, expecting benefits such as ACID transactions, efficient querying, and a single-file database. However, during the technical spike we discovered that the official am4 Python package (which provides an SQLite-backed API) crashes with a segmentation fault during initialization due to incompatibilities between its C‑extension, DuckDB, and PyArrow.
Decision
We decided not to use SQLite (via the am4 runtime package) for the MVP. Instead, we access the source Parquet files directly through a Dataset Service layer. This avoids the unstable dependency while still providing access to the same underlying data.
Consequences
- Positive: Eliminates the segmentation fault risk; simplifies dependencies (only pandas and a Parquet engine are required).
- Positive: Parquet files are the official source of truth; reading them directly ensures we use the exact data distributed by the AM4 package.
- Negative: Loss of SQL querying convenience; we implement ad‑hoc filtering in Python (acceptable for the MVP’s data volume).
- Negative: No built-in transactional guarantees; not needed as the data is read‑only.
Note: SQLite may be reconsidered in future versions for caching or user‑generated data, but for the MVP the direct Parquet approach is preferred.
