ADR-002: Why Direct Parquet Access Is Used
Status: Accepted
Context
The AM4 datasets are distributed as Parquet files within the am4 Python package. Using the am4 runtime package leads to a segmentation fault (see ADR-001). However, the Parquet files themselves are stable and can be read safely with standard libraries (pandas + pyarrow/fastparquet).
Decision
We decided to bypass the am4 runtime package entirely and read the Parquet files directly via a Dataset Service layer. The service loads each Parquet file into a cached pandas DataFrame on first access and provides typed getter methods for the rest of the application.
Consequences
- Positive: No more segfaults; the application starts reliably.
- Positive: Full control over data loading; we can add fallback mock data if files are missing.
- Positive: Parquet is column‑compressed and efficient for the analytical workloads we perform.
- Negative: We must re‑implement simple lookups that the am4 package provided (e.g., search by name). This is mitigated by the DatasetService API.
- Negative: We lose any future updates to the am4 package’s business logic (if any). The current MVP only needs the raw datasets.
