ADR-004: Why DatasetService Abstraction Exists
Status: Accepted
Context
The application needs to access three core Parquet datasets: aircraft, airports, and routes. Initially, analysis modules read these files directly using pandas. This approach scattered data‑loading logic throughout the codebase, making it hard to change the storage mechanism (e.g., switching to SQLite, adding caching, or providing mock data for testing).
Decision
We introduced a DatasetService singleton that centralizes all data access. The service lazy‑loads each Parquet file once, caches the DataFrame, and offers a clean API: get_aircraft(), get_aircraft_by_name(), get_airport(), get_airport_by_code(), get_route(), find_routes(). It also handles missing files gracefully by falling back to mock data.
Consequences
- Positive: No module reads Parquet files directly; all data access goes through the service.
- Positive: Easy to swap the underlying storage (e.g., to SQLite or a remote API) without touching analysis or UI code.
- Positive: Mock‑data fallback ensures the UI remains functional during development or in environments where the Parquet files cannot be read.
- Positive: Centralized logging and health monitoring of data loads.
- Negative: Slight indirection overhead (negligible for DataFrame-sized data).
- Negative: Developers must remember to use the service instead of importing pandas directly. This is mitigated by code reviews and linting.
